# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A collection of reference skills for LLM tooling, covering OGC standards and tooling. Skills follow the [Agent Skills](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview) format: a directory with a `SKILL.md` entry point (YAML frontmatter + Markdown body) and supporting `.md` files.

## Testing the site build locally

```bash
python3 .github/scripts/build_site.py _site <any-sha> <repo-url> <site-url>
# e.g.
python3 .github/scripts/build_site.py _site abc123 https://github.com/ogcincubator/ogc-llm-skills https://ogcincubator.github.io/ogc-llm-skills
```

This writes `_site/` with all skill zips, `index.html`, `manifest.json`, and `llms.txt`. No other build step exists; there are no tests.

## Publishing

Every push to `master` triggers `.github/workflows/publish.yml`, which runs `build_site.py` and deploys to GitHub Pages. No manual step needed.

## Adding a skill

1. Create a directory under a namespace (e.g. `bblocks/consuming/`).
2. Add `SKILL.md` with frontmatter:
   ```markdown
   ---
   name: kebab-case-name          # max 64 chars
   description: "What it covers and when to use it."   # max 1024 chars, plain text
   ---
   ```
3. Add supporting `.md` files in the same directory.

The zip name is derived from the directory path with slashes replaced by hyphens (`bblocks/authoring/` ‚Üí `bblocks-authoring.zip`).

## Skill content design principles

These are non-obvious and important for maintaining quality:

- **SKILL.md is a router, not a tutorial.** It must give enough signal per entry that the LLM can decide which sub-file to read without actually reading it ‚Äî closer to an annotated bibliography than a table of contents.
- **One lookup unit per file.** Each file answers a set of questions a developer would look up together. Target 400‚Äì1500 words. If two topics always need each other to be actionable, merge them.
- **Hierarchical navigation on demand.** The LLM reads `SKILL.md` first, then fetches sub-files via tool calls. Sub-files should be self-contained enough to be read independently.
- **Schemas by URL, not copied.** Config file schemas live at `https://raw.githubusercontent.com/opengeospatial/bblocks-postprocess/master/ogc/bblocks/schemas/` ‚Äî reference by URL so they stay current.

## Machine-readable entrypoints

The published site exposes two entrypoints for agent-driven installation:
- `manifest.json` ‚Äî structured JSON with absolute zip URLs for each skill, plus `commit` and `date` fields per skill for update-checking
- `llms.txt` ‚Äî Markdown summary following the [llmstxt.org](https://llmstxt.org) convention

Both are generated by `build_site.py` alongside `index.html`.

## What build_site.py injects into zips

`build_site.py` adds two things to each zip that are **not** present in the source skill directory:

- **`.version`** ‚Äî a JSON file with `commit`, `date`, `zip_url`, and `llms_txt` fields derived from the skill's last git commit. Do not create `.version` files in skill source dirs.
- **"Updating this skill" section** ‚Äî appended to `SKILL.md` inside the zip only. Do not add this section to source `SKILL.md` files.

Zips have **no top-level wrapper directory** ‚Äî files sit at the zip root, so extraction requires a pre-created target directory (`mkdir -p ~/.claude/skills/<name>` before `unzip`).

The recommended update strategy is **delete-and-replace**: remove the installed skill directory entirely and re-extract the new zip. Overwriting in place leaves stale files from renamed or deleted entries.
